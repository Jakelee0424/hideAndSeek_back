방탈출 게임의 서버 사이드 AI 동료 봇. 담당: 이재성(플레이어·이동 + AI 봇).

## 구조: 3겹

봇은 사람과 똑같이 "이동 의도(단위벡터)"만 만들고, 실제 이동·충돌·회전은 `Room.tick`이 사람과 공유
처리한다. **그래서 벽 회피는 공짜로 따라온다** — 봇 전용 경로탐색 코드가 없는 이유다.

```
빠른 층   BotBrain.steer()      매 tick(50ms). 목표로 향하는 단위벡터만. 절대 블로킹하지 않는다.
느린 층A  스크립트(reconsider)   항상 즉시 목표를 채우는 바닥. 안 풀린 퍼즐 중 최근접.
느린 층B  GroqBotPlanner        6초 주기, 가상 스레드. 성공하면 goal을 덮어쓴다.
```

핵심은 **B가 죽어도 게임이 안 멈춘다**는 것. `goal`이 volatile이고, LLM 실패(타임아웃·429·파싱실패·
없는 id 지어냄)는 전부 조용히 무시된다 → 봇은 스크립트 목표로 계속 움직인다. 우아한 열화가 설계의 축이다.

### 파일

| 파일 | 역할 |
|------|------|
| `BotBrain` | 3겹 조율. 도착 판정, LLM 호출 트리거, 결과 검증 |
| `BotPlanner` | 느린 층 인터페이스 (블로킹 허용) |
| `GroqBotPlanner` | Groq 호출 + 프롬프트 + 응답 파싱 |
| `BotContext` | 계획 시점 월드 사본(불변). 루프 스레드 → 플래너 스레드 |
| `BotProperties` | `game.bot.llm.*` 설정 |
| `Interactables` | 봇 목표 지점(POI) 레지스트리. 1단계 `Puzzles`를 대체 |
| `Goal` | `GOTO_PUZZLE` / `GOTO_NOTE` / `FOLLOW_PLAYER` / `IDLE` |

## 켜는 법

기본은 **꺼져 있다**(`game.bot.llm.enabled: false`). 켜두면 브라우저만 열어놔도 무료 한도가 샌다 —
`Room.join()`이 방마다 봇을 자동 스폰하기 때문.

```bash
# GROQ_API_KEY(gsk_...)는 Windows 사용자 환경변수에 둔다: setx GROQ_API_KEY "gsk_..."
# 없으면 Groq이 401을 뱉고 봇은 조용히 스크립트 층으로 폴백한다 → "LLM이 안 도는" 것과 구분이 안 된다.
# console.groq.com/keys 에서 발급. 기존 키 원문은 다시 볼 수 없으니 잃으면 새로 만들어야 한다.
./gradlew bootRun --no-daemon --args="--game.bot.llm.enabled=true --logging.level.com.game3d.server.game=DEBUG"
```

> **`--no-daemon`을 뺐다가 401이 나면 여기를 의심할 것.** Gradle 데몬이 살아 있으면 bootRun이 데몬의
> 환경을 물려받는다 — 키를 설정하기 *전에* 뜬 데몬이면 앱에 키가 안 들어간다.

DEBUG를 켜면 `봇 계획: GOTO_NOTE note-1 (문 코드 힌트 확인)` 처럼 모델의 판단 이유가 찍힌다.

> JDK 21 지정 빌드 필요: `-Dorg.gradle.java.home=$USERPROFILE/.jdks/ms-21.0.11`

## ⚠️ Groq 무료 티어 한도가 설계 제약이다

2026-07-15 실측(`x-ratelimit-*` 헤더). **모델 바꾸기 전에 반드시 다시 계산할 것.**

| 모델 | 요청/일 | 토큰/분 | 봇 1개 연속 가동 | 비고 |
|------|--------:|--------:|------------------|------|
| `openai/gpt-oss-20b` (**채택**) | 1,000 | 8,000 | **100분** | |
| `llama-3.1-8b-instant` | 14,400 | 6,000 | 약 20시간 | ❌ **2026-08-16 종료** |
| `llama-3.3-70b-versatile` | 1,000 | 12,000 | 100분 | ❌ **2026-08-16 종료** |

**모델 선택의 핵심**: 한도만 보면 `llama-3.1-8b-instant`가 14배 낫지만 **2026-08-16에 종료된다**
(무료·개발자 티어. 2026-06-17 공지, 대체안이 gpt-oss-20b). 그래서 한도를 포기하고 gpt-oss-20b를 택했다.
Groq 하면 떠오르는 게 llama라 무심코 되돌리기 쉬운데, 되돌리면 8/16에 봇이 죽는다.

6초 주기 = 10콜/분 → **1,000 요청/일 ÷ 10 = 하루 100분**이 전부다. 15분 플레이 기준 6~7판.
기본이 off인 이유가 이것이다. 토큰은 약 500/콜 × 10 ≈ 5,000 TPM으로 8,000 안쪽이라 여유 있다.
**주기를 줄이거나 방을 늘리면 요청 수가 먼저 터진다.**

## ⚠️ Spring Boot 4 = Jackson 3

Boot 4.0.7은 **Jackson 3.1.4(`tools.jackson.*`)** 를 쓴다. 2.x와 다르다:

- `com.fasterxml.jackson.databind`는 **클래스패스에 아예 없다** (annotations만 `com.fasterxml`에 남음)
- `JacksonException`은 **unchecked**다 (2.x `JsonProcessingException`과 다름) → `throws` 불필요
- `asText()`는 deprecated, **`asString()`** 으로 개명됨

HTTP도 Spring `RestClient` 대신 JDK `HttpClient`를 썼다. Boot 4의 관례 변화 리스크를 피하고
새 의존성도 필요 없다.

파싱 함정 하나: 모델이 `"targetId": null`을 주면 문자열 `"null"`로 읽힌다. `isNull()`로 명시적으로
걸러야 `IDLE`이 성립한다(`GroqBotPlanner.parseGoal`).

## 검증

`scripts/botcheck.mjs`. 30초간 STOMP 스냅샷을 받아 판정한다. 의존성 없음.

```bash
node --experimental-websocket scripts/botcheck.mjs 8081     # 포트 인자, 기본 8080
```

> `--experimental-websocket`은 Node 20에서 **필수**다(전역 WebSocket이 플래그 뒤에 있다). Node 22+면 불필요.

**판별 원리**: 스크립트 층은 `solvable=true`인 POI만 고르므로 `note-1`(쪽지)에 **절대 가지 않는다**.
따라서 봇이 note-1에 도달했다면 그 목표는 LLM이 낸 것(`GOTO_NOTE`)이다.

`FOLLOW_PLAYER`는 사람을 POI에서 9m+ 떨어진 구석(9,9)까지 걸어보낸 뒤, **정차한 다음부터만** 거리를 잰다.
걸어가는 도중을 포함하면 봇 스폰(0,0)과 사람 스폰(1,0)이 붙어 있어 봇이 가만히 있어도 1m가 찍힌다.

> 구 `botcheck3.mjs`는 레포 밖(`D:\tryThat`)에 뒀다가 **유실됐다**. 그래서 레포 안으로 옮겼다. 검증 도구를
> 레포 밖에 두지 말 것. (`GROQ_API_KEY`도 같이 날아갔다 — 환경변수뿐이라 백업이 없었다.)

### 2026-07-16 결과 (gpt-oss-20b, LLM on, 30초 × 2회)

| 항목 | 결과 |
|------|------|
| note-1 최소 접근 | **1.29m / 1.22m** ✅ → LLM 느린 층 동작 |
| LLM 호출 | **10건 전부 성공** (WARN 0건 — 400·파싱실패·무효 없음) |
| 계획 분포 | **10/10 `GOTO_NOTE note-1`** (다른 action 0건 — 아래 이슈 1) |
| 사람(정차 후) 최소 접근 | 9.81m / 10.12m → `FOLLOW_PLAYER` 미관측 |

**음성 대조**(`--game.bot.llm.enabled=false`)도 확인했다: note-1 6.19m·사람 12.49m로 두 판정 모두
정상적으로 '아니오'를 낸다 → 위 ✅가 스크립트 층 우연이 아님을 보증한다.

## 알려진 이슈 / 다음에 볼 것

0. ~~gpt-oss-20b 미실증~~ → **2026-07-16 해결.** 실증 완료(위 결과표). `max_tokens: 400` 수정도
   유효하다 — **10회 연속 400 `json_validate_failed` 없음**(120이던 시절엔 4회 중 1회 실패).
   `reasoning_effort: "low"`는 **넣지 않았다**. 400이 안 나는데 미리 넣을 이유가 없고, 추론 토큰을
   줄이면 판단 품질이 떨어질 수 있다. 400이 재발하면 그때 첫 후보로 쓸 것.

   단 표본 10회는 작다. 실패율이 낮아졌을 뿐 0이라고 단정할 근거는 아니다. 어차피 실패해도
   스크립트 폴백이라 게임은 안 멈춘다. 서버 로그의 WARN 빈도로 계속 볼 것.

1. 🚨 **봇이 note ↔ 퍼즐을 무한 반복한다. 최우선 과제다.** 2026-07-16 확증 — LLM 계획
   **10/10이 전부 `GOTO_NOTE note-1`**이었다. 추정이 아니라 측정된 사실이고, 8b만의 문제도 아니다
   (gpt-oss-20b도 똑같다). 루프의 정체:

   ```
   LLM: "힌트부터 확인해" → GOTO_NOTE note-1 → 봇 도착
     → 스크립트가 lastArrivedId(note-1)를 제외하고 최근접 퍼즐로 보냄
       → 6초 뒤 LLM이 또 GOTO_NOTE note-1 (아무것도 안 풀렸으니 판단이 바뀔 이유가 없다)
   ```

   **봇에겐 "쪽지를 이미 읽었다"는 상태가 없다.** `lastArrivedId`를 프롬프트에 넣어뒀지만 모델이
   무시한다. 프롬프트를 손보는 것보다 **읽은 쪽지를 봇 상태로 남기고, 읽은 뒤엔 note를 후보에서
   빼는 것**이 정공법이다(모델의 선의에 기대지 않는다).

   ⚠️ 스크립트 층에도 별개의 왕복이 있다. LLM을 끄면 봇이 lockbox-1 ↔ door-1을 무한 핑퐁한다
   (`nearestUnsolved`가 *방금 도착한 하나*만 제외하므로, 아무도 안 풀면 두 지점을 오간다).
   즉 왕복은 원인이 둘이다. LLM만 고쳐도 안 없어진다.

2. **`FOLLOW_PLAYER`는 여전히 실증되지 않았다.** 2026-07-16 재검증 — 사람을 (9,9)에 세워두니
   봇은 9.8~10.1m까지만 접근했다. 즉 **구 botcheck3의 "사람 최근접 0.64m"는 스폰이 붙어서 생긴
   착시가 맞았다.** 애초에 LLM이 `FOLLOW_PLAYER`를 한 번도 내지 않으므로(위 10/10 GOTO_NOTE)
   경로 자체가 안 돌고 있다. 이슈 1을 고쳐 계획이 다양해진 뒤에 다시 볼 것.
3. **`say`(대사)** — `Goal`에 자리만 잡아둔 상태. 인게임 채팅(조예원 담당)이 붙은 뒤에 추가한다.
   방탈출 봇의 존재감은 사실상 여기서 나온다.
4. **프론트 동기화** — `Interactables.java`의 id·좌표는 프론트 `game/interactables.ts`와 반드시
   일치해야 한다. 한쪽 바꾸면 양쪽 반영.
